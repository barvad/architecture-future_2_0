# Обоснование перехода на событийную архитектуру

## Текущее состояние

В текущем ландшафте компании «Будущее 2.0» основным интеграционным центром является legacy DWH на базе Microsoft SQL Server 2008 и ESB на Apache Camel.

Через DWH и Camel проходят:

* медицинские данные,
* финансовые операции,
* AI-интеграции,
* BI-аналитика,
* данные внутренних сервисов клиник.

Такой подход сформировал ряд архитектурных ограничений:

* высокая связанность систем,
* централизация бизнес-логики в DWH,
* сложность масштабирования,
* длительное формирование аналитики,
* высокая стоимость изменений,
* низкая скорость интеграции новых направлений бизнеса.

Кроме того, DWH используется одновременно:

* как operational integration hub,
* как аналитическая платформа,
* как источник бизнес-логики,
* как слой интеграции между системами.

Это приводит к:

* перегрузке SQL Server,
* сложным ETL,
* росту latency,
* ухудшению time-to-market,
* рискам отказа центрального узла.

---

# Ограничения текущей архитектуры Camel + DWH

## 1. Централизованная связанность

В текущей архитектуре:

* DWH является центральной точкой интеграции,
* большинство сервисов зависят от схем БД,
* изменения структуры данных влияют на множество систем.

Последствия:

* высокий coupling,
* сложные миграции,
* невозможность независимого развития доменов.

---

## 2. Синхронные интеграции

Многие процессы используют:

* synchronous API,
* ESB orchestration,
* batch ETL.

Это приводит к:

* cascading failures,
* высокой latency,
* плохой устойчивости к нагрузке.

---

## 3. Ограниченная масштабируемость

Legacy DWH плохо подходит для:

* сотен терабайт данных,
* realtime telemetry,
* IoT streaming,
* event-driven AI pipelines.

Вертикальное масштабирование SQL Server становится экономически неэффективным.

---

## 4. Медленная аналитика

Формирование отчётов занимает часы из-за:

* сложных ETL,
* большого количества трансформаций,
* batch-подхода.

Self-service аналитика практически невозможна.

---

## 5. Сложность подключения новых направлений

Каждое новое направление:

* требует изменений в DWH,
* требует изменений в Camel routes,
* усиливает coupling.

Это противоречит целям масштабирования бизнеса.

---

# Целевая событийная архитектура

В целевой архитектуре компания переходит к:

* domain-driven architecture,
* event-driven integration,
* asynchronous communication,
* distributed ownership данных.

Ключевым механизмом интеграции становится Event Platform:

* Kafka / Redpanda,
* Schema Registry,
* stream processing,
* event contracts.

---

# Почему выбран событийный подход

## 1. Слабая связанность доменов

Домены взаимодействуют через события, а не через:

* shared database,
* synchronous calls,
* прямые SQL-интеграции.

Например:

* Patient Domain публикует `PatientRegistered`,
* Fintech Domain подписывается на событие,
* Analytics Platform обновляет витрины,
* AI Platform запускает обработку.

При этом домены:

* не знают внутреннюю реализацию друг друга,
* могут развиваться независимо,
* имеют собственные модели данных.

---

## 2. Масштабируемость

Event-driven architecture позволяет:

* независимо масштабировать consumers,
* обрабатывать миллионы событий,
* распределять нагрузку по partition-ам.

Особенно это важно для:

* IoT telemetry,
* realtime analytics,
* AI pipelines,
* fraud detection.

---

## 3. Near-real-time обработка

Вместо batch ETL:

* события обрабатываются потоково,
* аналитика обновляется почти мгновенно,
* KPI пересчитываются continuously.

Это позволяет:

* сократить latency аналитики,
* ускорить реакцию бизнеса,
* поддерживать realtime dashboards.

---

## 4. Гибкость подключения новых доменов

Новый домен может:

* публиковать собственные события,
* подписываться на существующие события,
* не изменять существующие сервисы.

Например:

* подключение фармацевтической компании,
* нового региона,
* нового AI-сервиса.

Интеграция становится additive, а не disruptive.

---

## 5. Повышение отказоустойчивости

Асинхронная архитектура:

* уменьшает cascading failures,
* позволяет использовать retries,
* поддерживает DLQ,
* обеспечивает replay событий.

Система становится устойчивее к частичным отказам.

---

## 6. Улучшение observability

События формируют:

* audit trail,
* lineage,
* tracing бизнес-процессов.

Это особенно важно для:

* медицинских данных,
* финансовых операций,
* compliance.

| ------------------------- | -------------- | ------------------------- |
| Связанность               | Высокая        | Низкая                    |
| Масштабируемость          | Ограниченная   | Горизонтальная            |

| Отказоустойчивость        | Ограниченная   | Высокая                   |
| Гибкость изменений        | Низкая         | Высокая                   |

| Скорость изменений        | Низкая         | Высокая                   |
| Replay данных             | Ограничен      | Поддерживается            |

| Независимость доменов     | Низкая         | Высокая                   |

# Роль legacy систем в переходный период

Полный отказ от legacy DWH и Camel невозможен одномоментно.

Поэтому используется:

* Anti-Corruption Layer,
* CDC replication,
* event translation layer.

На переходном этапе:

* legacy системы становятся compatibility bridge,
* новые сервисы строятся event-native,
* бизнес-логика постепенно выносится из DWH.

---

# Результат трансформации через 3 года

Целевая архитектура позволит компании:

* масштабировать новые бизнес-направления,
* запускать новые регионы,
* подключать внешних партнёров,
* перейти к realtime аналитике,
* снизить coupling,
* ускорить time-to-market,
* повысить устойчивость платформы.

При этом:

* DWH перестанет быть центральным интеграционным узлом,
* Camel станет временным compatibility layer,
* домены получат независимость,
* аналитика станет self-service и near-real-time.
